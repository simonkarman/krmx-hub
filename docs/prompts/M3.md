# M3 — Game registry, provisioning, instance lifecycle, reaper

Read docs/ARCHITECTURE.md §5, §6.1, §6.4, §9.1/9.2/9.8/9.14.

Goal: a host registers a game; a user creates an instance; a local provisioner
spawns a server; dead servers get reaped.

Deliver:
- Host CRUD for game + game_version (frontend_url registered here, immutable).
- Instance creation flow: service token (opaque, sha256-hashed), HMAC-signed
  provision call (x-hub-timestamp + x-hub-signature), 60s timeout -> cancel.
- examples/tictactoe/provisioner: local HTTP server that verifies the HMAC,
  spawns a placeholder server process, returns ws://localhost:<port>.
- Service API: heartbeat/state endpoints with service-token auth.
- Reaper (interval in dev): stale heartbeat -> cancelled + token revoked
  (+ hold release wired in M4; leave a TODO seam).
- Security tests: P-01..P-06, S-01..S-05, S-07, L-09 (token/cancel part).

Done when: create-instance round trip works locally against the provisioner.
Review focus: invariants 1, 2, 8, 14 — provision response handling especially.
